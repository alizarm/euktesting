Table 1. Documentation Journey types and their icons
Icon Journey Type Description
Standalone Journey

A standalone journey consists of one workflow of articles to complete a specified task.
Super Journey

A super journey consists of a collection of workflows—also known as secondary journeys—that must be completed in their specified order to complete a task.
Pega Academy Super Journey A collection of workflows that must be completed in their specified order to coincide with Pega Academy missions.
Secondary Journey A secondary journey is a child journey of a super journey. Multiple secondary journeys make up one super journey.

Name Description

It advertises and promotes all the enterprise product offerings and solutions that Pegahas to offer, including Pega Platform™, the low-code platform for AI-powered decisioning and workflow automation. Academy Pega Academy is a learning resource that offers training and certifications for Pega Platform, a low-code development tool for enterprise applications.   • Self-study training: Online missions that eliminate travel costs and minimize time away from work.
  • Instructor-led training: Live programs taught by Pega-certified experts.
  • Online mentoring: Self-study training with guidance from a certified Pega Academy instructor.

Users can also ask questions, post discussions, and find answers to questions from other users. The Support Center also provides updates on security alerts, patch releases, resolved issues, and support documents. The Pega Support Center also includes the My Support Portal (MSP), allowing users to:
  • Submit support tickets.
  • Set, view, or remove notification preferences.
  • Share large files with the support team.
  • Review a summary of open tickets.
  • Download important materials as PDFs.

6.1.68 March 4 2025
Title
Populates errors in the correct context when there are errors
Notes
Populates errors in the correct context in the case of 5xx errors.

Previously, if there was a server error caused by an interaction in the work area context, the error was set to application context even though the context in the headers was set to work area. As a result, in the event of a 200 API call during a later interaction, the existing message was not getting cleared because of a context mismatch.

Following the change, the message is linked to the correct context, as defined in the header.
